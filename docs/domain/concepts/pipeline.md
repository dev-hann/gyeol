# Pipeline (파이프라인)

## 정의

파이프라인은 태스크가 **생성부터 완료까지 거치는 전체 실행 흐름**을 관리하는 개념입니다.

스케줄러(Scheduler), 큐(Queue), 메시지 버스(MessageBus) 세 컴포넌트가 협력하여 파이프라인을 구동합니다. 사용자는 이 세 컴포넌트를 직접 다루지 않고, 태스크를 제출하거나 스레드를 실행하는 방식으로 파이프라인에 참여합니다.

## 구성 컴포넌트

### 스케줄러 (Scheduler)

파이프라인의 오케스트레이터입니다. 다음 역할을 수행합니다:

1. **태스크 접수**: 외부에서 들어온 태스크를 큐에 등록
2. **태스크 분배**: 큐에서 태스크를 꺼내 레이어 레지스트리로 적절한 레이어를 찾음
3. **워커 실행**: 레이어에 소속된 워커들에게 태스크를 병렬로 분배
4. **결과 처리**: 워커 실행 결과로 파생된 후속 태스크를 큐에 재등록

동시 실행 제한: 최대 4개 태스크까지 병렬 처리 (`maxConcurrent`)

### 큐 (Queue)

태스크를 **우선순위 기반으로 정렬 보관**하는 대기열입니다.

- 높은 우선순위(High) → 보통(Medium) → 낮음(Low) 순서로 처리
- 같은 우선순위 내에서는 선입선출(FIFO)
- 이진 검색 기반 정렬 삽입으로 O(log n) 성능 보장

### 메시지 버스 (Message Bus)

태스크 이벤트를 구독자에게 전달하는 **발행/구독(Pub/Sub) 시스템**입니다.

- 특정 태스크 타입에 대한 구독: `subscribe("analysis_result", handler)`
- 와일드카드 구독: `subscribe("*", handler)` — 모든 이벤트 수신
- 용도: UI 갱신, 로깅, 모니터링 등

### 레이어 레지스트리 (Layer Registry)

모든 레이어의 등록부입니다. 입력 타입으로 레이어를 검색할 수 있습니다.

## 실행 모드

### 단일 실행 (runOnce)

큐에서 태스크를 꺼내 한 번 실행하는 모드입니다:

```
큐 → 최대 4개 태스크 꺼냄
     │
     ├→ 태스크 A → 레이어 매칭 → 워커 병렬 실행 → 결과
     ├→ 태스크 B → 레이어 매칭 → 워커 병렬 실행 → 결과
     ├→ 태스크 C → 깊이 초과 → 무시
     └→ 태스크 D → 매칭 레이어 없음 → 무시

결과의 후속 태스크 → 큐에 재등록 → MessageBus로 이벤트 발행
```

### 스레드 실행 (runThread)

하나의 스레드를 실행하는 모드입니다. 자세한 내용은 [Thread 개념 문서](thread.md)를 참조하세요.

### 전체 스레드 실행 (runAllThreads)

모든 활성 스레드를 순차적으로 실행하는 모드입니다. 비활성 스레드는 건너뜁니다.

## 실행 로그 (Execution Log)

모든 워커 실행의 성공/실패가 기록됩니다:

- 성공 시: 태스크 ID, 워커 이름, "success" 상태
- 실패 시: 태스크 ID, 워커 이름, "failed" 상태, 에러 메시지

실행 로그는 모니터링과 디버깅에 사용됩니다.

## 보안 장치

- **최대 실행 깊이**: 10 — 무한 파생 방지
- **동시 실행 제한**: 4 — 시스템 과부하 방지
- **도구 호출 루프 제한**: 5회 — 채팅에서 AI가 무한히 도구를 호출하는 것을 방지

## 다른 개념과의 관계

- **Task**: 파이프라인의 처리 대상. 큐에 들어가 스케줄러에 의해 실행됨
- **Layer**: 레이어 레지스트리에 등록되어 스케줄러가 태스크를 매칭할 때 사용
- **Worker**: 스케줄러가 태스크를 워커에게 전달하여 실제 처리를 수행
- **Thread**: 스레드 실행 시 파이프라인을 통해 레이어를 순차적으로 통과
- **Provider**: 워커 실행 시 제공자를 통해 LLM API에 접근
